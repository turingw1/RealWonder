# RealWonder 光流训练 Helios Pipeline（方案 B）

> 结合 `组会508.md`、`508 大组会.md`、`组会428讨论_细化.md` 和 `训练pipeline.md` 整理。  
> 本笔记只写方案 B：**RealWonder 直接迁移到 Helios，并通过微调让 Helios 学会光流控制**。

## 0. 目标

目标不是复现 RealWonder 原后端，而是保留 RealWonder 最核心的中间表示，把生成 backbone 换成 Helios：

```text
真实视频 / 物理仿真 / 交互动作
        |
        v
光流 F + 粗糙 RGB / depth / point cloud preview
        |
        v
Helios I2V / V2V / interactive generation
        |
        v
物理运动可控、视觉质量更好、支持长视频和交互
```

一句话概括：

> RealWonder 负责把动作或物理仿真翻译成“应该怎么动”，Helios 负责把这个运动条件渲染成真实视频。

这里真正要迁移的是 RealWonder 的后半段能力：

```text
光流 F + 粗糙 RGB Ṽ + 首帧 I + prompt -> 真实感视频
```

只不过原来这个后端是 Wan I2V / distilled diffusion，现在要换成 Helios。

## 1. 5 月 8 日后的明确方向

5 月 8 日组会里和方案 B 直接相关的信息：

- **Helios 可以 image-to-video**，所以可以从首帧 I 开始生成未来视频；
- **RealWonder 可以直接迁移过来但换成 Helios，需要微调一下**；
- **光流先用 RAFT**，不要一开始换复杂 flow 模型；
- **微调训练数据先用 MixKit Dataset**，快速把训练链路跑通；
- 物理交互最终要落到可控场景上；
- 当前背景重建问题很大，地板凹陷、墙壁弯曲，不符合物理；
- Hunyuan World mesh 太慢且物体不规则，后续更适合走 **点云 + 深度估计**；
- 之后可以搬到 A800/A100 服务器上跑训练和备份。

因此本阶段核心问题是：

> 如何让 Helios 学会读取 RealWonder 式的光流和粗糙结构条件。

## 2. 总体 Pipeline

建议把方案 B 拆成四层：

```text
Layer 1: video-edit 自监督训练
真实视频 -> RAFT flow + coarse preview -> Helios 重建原视频

Layer 2: Helios 光流控制微调
冻结 Helios 主体，训练 flow/coarse adapter 或 LoRA

Layer 3: RealWonder / Genesis 物理迁移
单图 -> 3D/点云/Genesis -> flow + coarse preview -> Helios 生成

Layer 4: 长视频与交互
Helios chunk autoregressive generation + action signal / flow chunk 对齐
```

这样做的原因是：先让 Helios 在真实视频上学会“听光流”，再把输入源从真实视频 RAFT flow 换成 RealWonder / Genesis flow。

不要一开始就依赖物理仿真数据，因为那会把问题绑在场景重建、材料估计、仿真稳定性上，训练链路很难 debug。

## 3. 数据 Pipeline

### 3.1 第一阶段：MixKit 小规模跑通

5 月 8 日提到 MixKit，适合作为第一阶段验证集。

数据来源：

```text
LanguageBind/Open-Sora-Plan-v1.1.0/all_mixkit
```

处理流程：

```text
mp4
  -> 抽帧 clip
  -> RAFT 提取 flow_fwd
  -> 保存首帧 I
  -> 生成 coarse_rgb / degraded video
  -> 读取或生成 prompt/caption
  -> 得到 Helios 训练样本
```

建议第一版规模：

- 100 到 500 条 clip；
- 每条裁成 Helios 单 chunk，例如 33 帧；
- 分辨率跟 Helios 官方 I2V/V2V 配置对齐；
- 每条样本都保存 flow 可视化，先肉眼确认 RAFT 没崩。

第一阶段的目标不是训练出好模型，而是确认：

- Helios 环境能跑；
- RAFT 预处理能跑；
- 数据格式能被 Helios dataloader 正确读取；
- adapter / LoRA 可以参与训练；
- 10 条视频可以被过拟合。

### 3.2 第二阶段：扩大到 OpenVid 或更大视频集

沿用 4 月 28 日讨论里的思路：把 RealWonder 的训练改造成自监督 video-edit / video-reconstruction 任务。

```text
V_src
  -> RAFT / GMFlow 提取光流 F
  -> first-frame forward warp 或 blur-downsample 得到 coarse_rgb
  -> Helios 在 (I, F, coarse_rgb, prompt) 条件下重建 V_src
```

这一步要学到的通用能力是：

> 给定首帧和运动场，Helios 能生成一段沿着该运动场演化的视频。

这个能力训练出来后，RealWonder / Genesis 输出的 flow 才能作为替代输入接进来。

### 3.3 每条样本保存字段

建议不要只保存 latent，必须保留原视频和 raw flow，方便之后换 Helios 配置或重新编码。

```text
video_id/
  frames.mp4 或 frames.npy
  first_frame.png
  flow_raw.npz          # [T-1, H, W, 2]
  flow_vis.mp4          # debug 可视化
  coarse_rgb.mp4        # blur/downsample 或 first-frame forward warp
  depth.mp4             # 可选，后续接点云/场景
  mask.mp4              # 可选，目标物体区域
  prompt.txt
  meta.json             # fps, resolution, crop, scale, flow scale
```

关键原则：

- `flow_raw.npz` 是主输入，后续 adapter 最好直接读 raw tensor；
- `flow_vis.mp4` 只用于检查和可视化，不作为唯一训练输入；
- `coarse_rgb` 第一版可以简单做 blur/downsample，第二版再做 first-frame forward warp；
- `frames` 必须保留，避免 Helios 的 VAE、temporal stride、分辨率变化后无法重编码。

## 4. Helios 条件注入方式

Helios 的生成范式和 RealWonder 原 Wan 后端不一定一样，所以不应该直接假设 flow-warp noise 可以照搬。建议按从简单到复杂推进。

### 4.1 Baseline：只用 Helios I2V / V2V

先跑最小 baseline：

```text
first frame I + prompt -> Helios I2V -> video
```

或者：

```text
coarse_rgb / degraded video + prompt -> Helios V2V -> video
```

这一步不算最终方案，只是确认 Helios 自身能力：

- 首帧外观能否保持；
- prompt 能否跟随；
- 短视频能否延长；
- 单 chunk 和多 chunk 是否稳定。

### 4.2 推荐主线：冻结 Helios，训练小 control adapter

主线是冻结 Helios backbone，只训练轻量条件模块：

```text
I, prompt, flow_raw, coarse_rgb/depth
        |
        v
flow/coarse encoder
        |
        v
Helios control adapter / LoRA
        |
        v
Helios output video
```

可选结构：

1. **通道拼接型 adapter**
   - 把 `flow_raw` 或 `flow_vis` 编码到视频 latent 同尺度；
   - 把 `coarse_rgb` 编码到 latent 同尺度；
   - 通过 1x1/3D conv 或 MLP 投影；
   - 和 Helios latent concat 或 residual add。

2. **Cross-attention control token**
   - raw flow `[dx, dy]` 加时间和空间位置编码；
   - flow encoder 输出 control tokens；
   - 插入 Helios block 的 cross-attention K/V。

3. **LoRA + adapter**
   - Helios 主体冻结；
   - LoRA 加在 q/k/v/o 或 temporal attention；
   - adapter 负责把 flow/coarse preview 转成 Helios 可读条件。

建议第一版：

```text
冻结 Helios backbone
训练 flow/coarse encoder + LoRA
只做 33 帧单 chunk
用 MixKit 做 10 条视频 overfit
```

如果这个 overfit 都做不到，说明条件注入位置、loss 或数据格式有问题，不要急着扩大数据。

### 4.3 是否保留 RealWonder 的 flow-warp noise

需要看 Helios 的底层范式：

- 如果 Helios 仍然有 diffusion / rectified flow 路径，可以尝试把 RealWonder 的 flow-warp noise 迁移过来；
- 如果 Helios 更接近 autoregressive latent/video token prediction，则应该把 flow 作为 control token 或 condition embedding，而不是强行套 noise warp；
- 如果 Helios 有 V2V 编辑接口，可以先用 coarse_rgb 做输入，再用 flow adapter 加强运动控制。

结论：

> flow-warp noise 是 RealWonder 的重要思想，但迁移到 Helios 时不是必须第一步照搬。第一优先级是让 Helios 通过 adapter 学会稳定服从 raw flow。

## 5. 训练 Pipeline

### Stage 0：Helios 接口检查

先检查并记录：

```text
字段 | Helios 配置 | 对训练的影响
VAE latent channel
temporal stride
chunk size
输入帧数
I2V 输入格式
V2V 输入格式
hidden dim
block 数
attention 接口
KV cache / autoregressive state
```

同时跑通：

- Helios 官方 I2V；
- Helios 官方 V2V 或 interactive demo；
- 单 chunk 生成；
- 多 chunk 生成；
- 输出视频保存和评测脚本。

### Stage 1：MixKit 单 chunk 过拟合

目标：确认训练链路正确。

```text
MixKit clip
  -> 33 frames
  -> RAFT flow
  -> coarse_rgb
  -> Helios adapter
  -> reconstruct original clip
```

训练设置：

- 冻结 Helios；
- 只训练 control encoder + LoRA；
- 先 overfit 10 条视频；
- prompt 可以先固定成简单 caption；
- coarse_rgb 第一版用 blur/downsample；
- 第二版再换成 first-frame forward warp。

验收标准：

- loss 能下降；
- 输出视频不是纯靠 prompt 幻想；
- `no-flow` 时运动明显变弱；
- `reversed-flow` 时运动方向能改变；
- 只给 flow 不给 coarse_rgb 时，遮挡和结构会变差；
- 加 coarse_rgb 后结构和遮挡更稳。

### Stage 2：扩大到 MixKit / OpenVid

目标：训练通用光流控制能力。

```text
V_src -> (I, F, coarse_rgb, prompt) -> Helios -> V_pred
```

训练目标根据 Helios 范式决定：

- diffusion / flow matching：预测 velocity 或 noise；
- autoregressive latent：预测下一段 latent；
- video token：预测下一段 token；
- V2V 编辑：重建 loss + temporal consistency。

建议加入辅助 loss：

- 生成视频再提 RAFT，与输入 flow 算 EPE；
- 对目标区域 mask 单独算 flow consistency；
- chunk 边界帧加 temporal consistency；
- prompt/CLIP loss 只能作为辅助，不能压过 flow；
- 可以加 LPIPS / perceptual loss 保视觉质量。

### Stage 3：接入 RealWonder / Genesis flow

这一步才把输入源从真实视频换成物理仿真：

```text
单图 I
  -> 点云/depth/3D重建
  -> Genesis 或物理规则
  -> flow_raw + coarse_rgb/depth preview
  -> Helios adapter
  -> final video
```

第一批测试沿用已有 12 个物理 case，先选其中 3 到 6 个：

- 平移/碰撞；
- Jenga push；
- cloth draping；
- cushion squeeze；
- liquid water vs honey；
- prompt vs simulation adversarial。

核心验证：

- 用真实视频训练出的 flow adapter，能不能泛化到 Genesis flow；
- prompt 会不会覆盖物理运动；
- 只有 flow 是否不够，是否必须加入 coarse_rgb/depth；
- 接触、遮挡、新露出区域是否稳定。

### Stage 4：Helios 长视频和交互

Helios 的优势是长视频和交互，因此训练后要做 chunk 化控制：

```text
flow chunks: 33 + 33 + 33 + ...
first frame / previous chunk
        |
        v
Helios autoregressive generation
```

需要处理：

- flow/control condition 按 Helios chunk 对齐；
- chunk 间做 overlap，避免边界跳变；
- KV cache 或 hidden state 是否保留；
- 长视频 drift；
- 每秒接受一次 action signal；
- 动态场景可考虑 router，把背景、物体、交互区域分开处理。

## 6. 背景和物理交互方案

5 月 8 日提到背景重建目前是硬问题：

- 地板完全凹陷；
- 墙壁弯曲；
- 场景不符合物理规则；
- Hunyuan World mesh 生成慢；
- mesh 里的物体形状不规则。

因此建议短期不要把 Helios 光流训练绑定到完整 3D mesh 上。

更稳的拆法：

```text
短期：
  真实视频 -> RAFT flow
  先训练 Helios 的光流控制能力。

中期：
  depth estimation + point cloud
  生成 coarse preview、object mask、背景约束。

长期：
  物体和背景分离
  动作只作用在可交互物体
  背景用点云/depth/camera alignment 保持稳定。
```

场景输入可以逐步变成：

```text
RGB image
  -> depth
  -> point cloud
  -> object segmentation
  -> interaction region
  -> physics update / projected flow
  -> Helios generation
```

这样 RealWonder 的物理部分和 Helios 的视频生成部分是解耦的，任何一边都可以单独替换和评测。

## 7. 评测指标

评测不能只看画质，要围绕“是否听光流、是否听物理”。

### 7.1 光流一致性

```text
input flow F
generated video -> RAFT -> F_pred
EPE(F, F_pred)
```

最好在 object mask 内单独计算一次。

### 7.2 轨迹一致性

用 SAM / tracker / keypoint 跟踪生成视频中的目标物体，和 Genesis 投影轨迹比较像素偏差。

### 7.3 事件时序

比较关键事件发生帧：

```text
碰撞 / 落地 / 倒塌 / 接触 / 穿插
Genesis event frame vs generated event frame
```

### 7.4 Prompt 覆盖测试

做对抗 prompt：

```text
flow: object moves right
prompt: object moves left
```

如果视频跟 prompt 左移，说明 Helios 没有可靠服从 flow。

### 7.5 长视频稳定性

- chunk 边界是否跳变；
- 后 100 帧是否漂移；
- 背景是否越来越弯；
- 目标物体身份是否保持；
- action signal 变化后是否能及时响应。

## 8. 决策树

```text
Helios 官方 I2V / V2V 能跑
        |
        v
MixKit 10 条视频 overfit
        |
        |-- overfit 失败
        |       -> 检查 adapter 注入位置、loss、flow scale、数据格式
        |
        |-- overfit 成功
                -> 扩大到 500-5000 条 clip
                -> 加 no-flow / reversed-flow ablation
                -> 加 flow consistency loss
                |
                v
        接入 Genesis / RealWonder flow
                |
                |-- Genesis flow 泛化差
                |       -> 做 flow scale normalization
                |       -> 加 synthetic flow / physics flow finetune
                |       -> 加 coarse_rgb/depth
                |
                |-- Genesis flow 可用
                        -> 做长视频 chunk 控制
                        -> 接 action signal
                        -> 做物理交互 demo
```

## 9. 近期任务

### Week 1：Helios 和数据链路跑通

- 跑 Helios 官方 I2V；
- 跑 Helios 官方 V2V 或 interactive demo；
- 下载并整理 MixKit 小样本；
- 用 RAFT 处理 3 到 5 个 MixKit 视频；
- 保存 `flow_raw.npz`、`flow_vis.mp4`、`coarse_rgb.mp4`；
- 输出一组数据可视化检查。

### Week 2：最小 adapter 训练

- 确认 Helios chunk/frame/latent 格式；
- 实现 `MixKit -> flow_raw + coarse_rgb -> Helios` dataloader；
- 实现最小 control adapter；
- 冻结 Helios，训练 10 条视频 overfit；
- 做 `flow / no-flow / reversed-flow` ablation。

### Week 3：扩大训练

- 扩到 500 到 5000 条 MixKit/OpenVid clip；
- 加 flow consistency loss；
- 加 coarse_rgb/depth ablation；
- 比较只用 I2V、I2V+flow、I2V+flow+coarse_rgb 三种设置。

### Week 4：接 RealWonder / Genesis

- 从现有物理 case 导出 `flow_raw + coarse_rgb/depth`；
- 接到 Helios adapter；
- 跑 3 到 6 个物理测试 case；
- 记录光流一致性、轨迹一致性、事件时序；
- 判断是否需要 physics-flow finetune。

### Week 5：长视频和交互

- 做 3 个 chunk 以上长视频；
- 测 chunk 边界和 drift；
- 接每秒一次 action signal；
- 初步做物理交互 demo。

## 10. 一句话方案

> 用 MixKit/OpenVid 从真实视频中提 RAFT 光流，把 RealWonder 的“光流 + 粗糙 RGB”后端训练任务改成 Helios 的 video-edit/self-reconstruction 任务；先冻结 Helios 训练轻量 flow/coarse adapter 和 LoRA，让 Helios 学会服从 raw optical flow，再把输入源从真实视频切换到 RealWonder / Genesis / 点云深度产生的物理光流，最终实现可交互、可长视频生成的 Helios 物理世界模型。
